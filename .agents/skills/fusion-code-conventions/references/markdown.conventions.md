# Markdown Code Conventions

Conventions for authoring Markdown across all contexts — documentation, READMEs, changelogs, skill files, and any prose-heavy file.

> **Applicability:** These are org-wide baseline defaults. Repository-level policy (`CONTRIBUTING.md`, ADRs, contributor guides) and tooling configuration (`.editorconfig`, `.markdownlint.json`) take precedence when they explicitly override a rule below. See the skill's **Precedence and applicability** section for the full resolution order.

Reference: [CommonMark spec](https://spec.commonmark.org/) · [GitHub Flavored Markdown spec](https://github.github.com/gfm/)

---

## Frontmatter

- Must be valid YAML: no malformed keys, unquoted special characters, or broken indentation.
- Delimited by `---` on the first line and a closing `---` before the document body.
- String values containing `:`, `#`, `{`, `}`, `[`, `]`, or leading/trailing spaces must be quoted.
- Use `null` for explicitly empty values; omit the key entirely if it is not applicable.

## Document structure

- One `#` (H1) per document — the document title.
- Do not skip heading levels (`##` → `####` is invalid).
- No duplicate heading text at the same nesting level within a section.
- Blank line before and after headings, fenced code blocks, and block quotes.
- Heading anchors are auto-generated by most renderers: lowercase, spaces → `-`, punctuation removed. Update all `[text](#anchor)` links when headings change.

## Links

- Prefer relative links for files in the same repository. Relative links survive clones and forks; absolute URLs do not.
- Internal anchor links (`[text](#anchor)`) must target an existing heading.
- No bare URLs in prose — wrap in `<>` or use `[label](url)`.

## Code blocks

Every fenced code block must have a language identifier. Common: `ts`, `tsx`, `js`, `bash`, `sh`, `yaml`, `json`, `csharp`, `md`, `text`.

````markdown
```ts
const x = 1;
```
````

Use single backticks for inline code and commands: `` `git status` ``.

## Lists

- Use `-` for unordered lists; be consistent within a document. (`*` and `+` are valid but mixing styles reduces readability.)
- Ordered lists start at `1`.
- Nest by aligning the list marker under the first character of the parent item's text.

## Emphasis

| Intent | Syntax |
|---|---|
| Bold | `**text**` |
| Italic | `*text*` |
| Bold + italic | `***text***` |

Do not mix `**`/`__` or `*`/`_` styles within the same document.

## Images

Every image must have descriptive alt text: `![alt text](url)`.
Use relative paths for images stored in the repository.

## Callouts

Portable callout style that works in any renderer:

```markdown
> **Note:** Informational note.

> **Warning:** Urgent issue requiring attention.
```

On GitHub, prefer the native alert syntax instead (see [GitHub-specific → Alerts](#alerts)).

## Formatting

- Spaces (not tabs) for indentation inside lists and code blocks.
- No trailing whitespace. (Two trailing spaces create a hard line break in some renderers, but a blank line is clearer and universally supported.)
- Prose lines under 120 characters for diff readability; code blocks may exceed this.

## GitHub-specific

Rules that apply when the render target is GitHub (GFM).
Reference: [GitHub basic syntax guide](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)

### Alerts

Use GFM alert syntax instead of plain blockquotes. GitHub renders these with distinctive icons and colours.

```markdown
> [!NOTE]
> Informational note.

> [!TIP]
> Optional helpful advice.

> [!IMPORTANT]
> Key information required to succeed.

> [!WARNING]
> Urgent issue requiring immediate attention.

> [!CAUTION]
> Risk or negative consequence of an action.
```

Limit to one or two alerts per document. Do not nest them.

### Strikethrough

`~~text~~` renders as ~~strikethrough~~. This is GFM-only and not supported in standard CommonMark.

### Task lists

```markdown
- [x] Completed item
- [ ] Incomplete item
```

### Mentions and references

- Mention a person or team with `@username` or `@org/team`.
- Reference an issue or PR with `#123` or a full URL.
- Close an issue from a PR body or commit message with a closing keyword: `Closes #123`, `Fixes #123`, `Resolves #123`.
